The Cluster Control module provides a simple interface to most of the functionality provided by the command-line server tools, such as control_cluster.py, bot_op.py, and so on.

It allows users to perform common tasks associated with running servers, such as:

  • Starting and stopping the server
  • Inspecting the status of machines and server processes as they are running
  • Determining which other users are running servers on the same machines as you

1. Managing servers

The Cluster Control module's home page is the Manage Servers page.

When the server is running, the page displays information about the server processes, and provides links to start more processes, restart or stop the server, save the server layout for future startups, and view server logs. When the server is not running, a link to start it is displayed.

Each process also has an attached Actions menu. This is a context-sensitive list of capabilities for each process, similar to the right-click context menu that a standalone application would display for a particular item or object.

For most server processes, the actions are simply stopping, killing and restarting the process, and viewing the logs for it. CellApps and BaseApps have two additional links to connect to the process's interactive Python server and to reload Python modules related to entities and custom data types for the selected process.

This is the ideal place for customers or third parties to provide their own capabilities and extensions. Starting points for implementing extra capabilities are:

  • bigworld/tools/server/web_console/common/caps.py
  • The Exposed class in bigworld/tools/server/pycommon/cluster.py

1.1. Starting the server

Following the Start the server link from the Manage Servers page causes WebConsole to display the server's environment settings, and provides options as to which machines the server components will be started on.

The Server Environment table at the top of the page displays information about the user you will be starting the server as. In particular, it displays the BigWorld environment settings that will be provided to each server process at startup (MF_ROOT and BW_RES_PATH). These values are read from the file ~/.bwmachined.conf in the user's home directory, and if an inconsistency in these values is detected between the various machines being used to start processes, an error will be raised.

The Start table provides different ways to select which machines will be used to run the server processes. The simplest method is starting all processes on a single machine. Alternatively, users can select to start the processes on a particular group of machines. Machine groupings are defined in the file /etc/bwmachined.conf. The group menu also includes an option to start server processes on all machines in the cluster.

Users can also opt to start server processes on machines as specified in a pre-defined layout. At present, these layouts can only be generated by saving the layout of a running server, using the Save this server layout link in the Manage Servers page.

If you would like the ability to hand-craft these layouts, please see the save and load commands supported by control_cluster.py, and examine the XML file format supported by those commands.

As the server starts up, a table will be displayed indicating how many processes of each type are dead (i.e. process isn't running), running (process exists but hasn't registered with bwmachined yet) and registered (running and registered with bwmachined). If at any point bwmachined fails to reply with info about a particular process, that process's counters will be replaced with question marks (?) until bwmachined replies with proper information.

1.2. Restarting The Server

Restarting the server via the WebConsole behaves identically to `cluster_control.py restart`, i.e. it will try to restart the server in the currently running layout, however if the current layout is missing any basic server processes (e.g. all CellApps have died for some reason), they will be automatically included in the layout before restarting.

2. Browsing machines

The All Machines page displays information about all machines on the network that are currently running bwmachined. Various runtime and hardware diagnostics are displayed for each machine. Clicking a machine brings a list of all processes running on it (under any UID).

3. Browsing users

The All Users page displays lists of both active (running at least one process that registers with bwmachined) and inactive (not running any processes that register with bwmachined) users. Clicking a user takes you to the Manage Servers page for that user, which is a handy way to administer servers under UID's other than your own.

At the bottom of the Users page is a link Refresh User List, which causes all bwmachined's on the network to drop and then reload their user mapping. This may be needed in the situation where a new user has placed a .bwmachined.conf file in their home directory but has not been detected by any bwmachined process.

4. Browsing watcher values

Watcher values are internal parameters of a running BigWorld server process that are explicitly exposed via the Watcher interface. Basically, watcher values allow users and applications to inspect their internal state.

Watcher values are organised hierarchically into a tree structure. You can browse to the root of a process's watcher tree by clicking its name anywhere in the Cluster Control module. The two main places where these links are available are:

  • The Process list on the Manage Servers page
  • The Process list of a particular machine (see section Browsing Machines above)

When browsing watcher values, each page displays the values and subdirectories for a particular Watcher directory. Some watcher values are readable and writeable, and some are read-only (information about the write permissions is not currently part of the network protocol that the Python tools use to connect to the server processes).

The Advanced Options check box at the bottom of the watcher values page will reveal the Actions menu for each watcher if it is selected. A writeable watcher value can be modified by selecting the Edit menu item in the Actions menu of the watcher and entering a new value on the ensuing page. Doing this for values with read-only access will have no effect.

For further details on watchers, see the Server Programming Guide, chapter The Watcher Interface.

5. See also

Most or all of the functionality provided by the Cluster Control module is also provided by the control_cluster.py shell utility. It also provides a number of other commands that are not supported by the WebConsole interface.

For a complete list of available commands, use the --help switch.

6. Caveats

None of the Bots functionality provided by bot_op.py has been integrated into the web interface. All management of bots must still be done via the console.
