The Log Viewer module provides an interface to the server message logs. It can be used to perform static searches through log data, as well as provide live views of log activity. Summaries of per-user log size and other information can also be generated.

This module replaces the CGI script bw_log_viewer.cgi from BigWorld Technology version 1.7 and earlier.

1. Displaying output

If you just want to start using the tools and see some server output, click the Live Output link in the left-hand side navigation menu (this assumes you are currently running a server). The output from the server will be continuously updated in the log window.

The default filter settings on the Search page will also give you the log output for the currently running server. Just click the Fetch button to run the query.

2. Searching

Most of the functionality of the Log Viewer module is provided by the Search page. It provides a set of filters with which users can restrict the log results to be displayed, as well as check boxes that control the format of the output. All settings can be saved and restored to repeat common searches at a later time.

Log results can be displayed as static searches or as continuously updated live output, using the Fetch and Live Output buttons, respectively.

2.1. Log filters

The table of filters at the top of the page allows the user to restrict the kinds of log results that will be returned by a query. Fields that are regular-text entry widgets, if left blank, will not be used to filter results.

Where appropriate, filters have a Show check box that controls whether that field will be displayed in the log output. A filter can still be used for filtering out results, even if it is not displayed.

The following filters are available:

Filter Description
Time Time range over which the logs will be searched. The time range is defined in a point-offset manner – the Time control is used to enter the time the user is interested in, and the Period is used to designate the period and direction the log should be searched in, relative to the nominated time.

Time can be entered as either a literal time, or as one of a set of supported meta-times.

If it is entered literally, then it must follow the exact datetime format that is seen in the log output (e.g., "Mon 09 Oct 2006 17:18:11.217" – the milliseconds at the end are optional, everything else is mandatory).

Clicking the small calendar icon on the right displays a calendar widget that can be used to easily enter times in the correct format.

All supported meta-times can be selected by clicking the images to the right of the text entry field. Currently supported meta-times are:

  • the beginning of the log
  • the time of the last server startup
  • the time of the last occurrence of a particular message
  • present time
Period Period is defined by both a magnitude and direction – the first two fields are used to enter a multiple of common time units (seconds, minutes, hours, or days) and the third field is used to indicate the direction (forwards, backwards, either side, to beginning, or to present).

When to beginning or to present is selected, the first two fields are disregarded.

NOTE: If there are too many log results to display at once, they will be split onto multiple pages, much like a Google search. The results on each page are always displayed in chronological order (earliest results at the top), however, the ordering of the pages is determined by the direction of the search:

  • When Period is forwards, either side, or to present, then result pages are returned in chronological order.
  • When Period is backwards or to beginning, then result pages are returned in reverse order (most recent result page first).
Machine Log results can be displayed for all machines, or for a single machine.

Currently, it is not possible to filter based on a substring of the hostname or a netmask.

Note that only machines that have sent at least one log message will appear in the filter list; not all machines running bwmachined will necessarily appear.

Username Log results are displayed for only one user at a time. As with the list of machines, only users who have sent at least one log message will appear in this list.
PID Standard UNIX PID assigned by the operating system.
AppID Incremental counter assigned by some of the server "manager" components to the processes they manage (e.g., cellapp01, baseapp02, etc...).
Process Processes to be displayed.

More than one selection can be made, by using Shift or Ctrl, or by drag selecting.

Severity Log levels to be displayed.

More than one selection can be made, by using Shift or Ctrl, or by drag selecting.

Message Text to pattern match against the log message itself. Two text fields are provided to allow the user to specify both inclusion and exclusion patterns.

This filter has three independent options:

Option Description
Case-Sensitive Toggles case-sensitive pattern matching.
Regex Match Toggles use of regular expressions (for details on regular expressions, see `man 7 regex`).
Pre-Interpolate Controls whether format strings (i.e., %s, %d, etc...) are interpolated before or after the filter stage of the query.

If you are not familiar with format strings or printf(), then you should leave this option checked. When checked, all log entries have their format strings interpolated before filtering takes place.

For example, if a server component calls INFO_MSG( "Foo %d", 12 ) and interpolate is on, then a search for the string '12' will be successful. If interpolate is off, then the same search would not match that particular entry.

Turning interpolate off results in faster queries, but may omit some wanted search results if you do not understand this mechanism.

Also, be aware that messages with severity SCRIPT and higher are stored as "%s", so you will not be able to match against any text from these messages unless pre-interpolate is switched on.

2.2. Query modes

The Log Viewer module supports running queries in either static or live mode:

  • A static query (triggered with the Fetch button) will not update with new results that might become available after its execution.

    There is a hard-coded limit to the number of results (10000 lines) that can be returned at one time; results may be paged through if more than this amount is returned from the query.

  • Live queries continuously update as new entries are written to the logs, much like calling `tail -f` on a log file.

    The Time and Period controls are ignored in live mode – when live mode is activated, the last 20 lines of output from the last hour are immediately displayed, and then new results are displayed as they are written to the logs.

2.3. Lines of context

Lines of context around matching results can be displayed. This works in exactly the same way as `grep -C`, and separates chunks of contiguous context with a -- line.

2.4. Saved filter settings

The Action drop-down list box located at the bottom right of the Log Filters table is used for saving, restoring, and deleting filter settings.

This is useful for saving common searches, and the default filter settings can be set here too.

The Action drop-down list box also allows users to manually send a message to the log. The message will be logged as sent from the Tools process.

2.5. Conserving screen real estate

The Log Filters table can be rolled-up by clicking the up arrow icon in the top right corner – this is especially useful on smaller displays.

If the Auto-Hide Settings check box at the bottom of the table is checked, then the table automatically hides itself when either the Fetch or Live Output button is clicked.

3. Live Output page

The Live Output page is a simplified version of the search page running a live output query, with no filters applied, and with the log output for the currently logged in user continuously displayed.

4. Usage Summary page

The Usage Summary page provides an overview of the size and period of each user's log.

Clicking a particular username takes you to a page displaying detailed statistics on each segment of that user's log.

5. See also

The bigworld/tools/server/message_logger folder contains (amongst other things) a collection of utilities that allow you to work with message logs stored on a locally-accessible file system in a similar manner to standard UNIX shell utilities.

The following are currently provided:

  • mlcat.py
  • mlls.py
  • mltar.py
  • mlrm.py

Refer to the --help output for details on each utility.
